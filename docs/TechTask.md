# **Платформа генерации сертификатов Decentrathon**

### **1\. Цель**

Разработать платформу для автоматической выдачи сертификатов участникам хакатона.

Логика:

* пользователь вводит свой e-mail  
* система проверяет, есть ли этот e-mail в загруженной базе  
* если e-mail найден, система берет имя из таблицы  
* подставляет имя в шаблон сертификата  
* генерирует сертификат  
* пользователь скачивает его

  ### **2\. Состав платформы**

Платформа состоит из двух частей:

**1\. Пользовательская часть**

* страница ввода e-mail  
* проверка e-mail  
* генерация сертификата  
* скачивание сертификата

**2\. Админ-панель**

* вход по логину и паролю  
* загрузка шаблонов сертификатов  
* настройка шаблонов  
* загрузка базы участников  
* управление выдачей сертификатов

  ### **3\. Пользовательская часть**

  #### **3.1. Основной сценарий**

Пользователь:

* заходит на сайт  
* вводит e-mail  
* нажимает кнопку “Получить сертификат”

Система:

* проверяет наличие e-mail в базе  
* если e-mail найден, берет связанное имя  
* генерирует сертификат  
* дает скачать файл

  #### **3.2. Что должно быть на странице**

* поле для ввода e-mail  
* кнопка “Получить сертификат”  
* сообщение о результате  
* кнопка скачивания сертификата после успешной генерации

  #### **3.3. Сообщения системы**

Если e-mail найден:

* “Сертификат готов”

Если e-mail не найден:

* “Данный e-mail не найден в базе участников”

Если выдача отключена:

* “Выдача сертификатов еще не открыта”

Если ошибка:

* “Произошла ошибка. Попробуйте позже”

  ### **4\. Админ-панель**

  #### **4.1. Авторизация**

Админ входит через:

* логин  
* пароль

Доступ в админ-панель должен работать через **JWT token authentication**.

#### **4.2. Управление шаблонами**

Админ может:

* загрузить шаблон сертификата  
* выбрать активный шаблон  
* удалить или заменить шаблон

Поддерживаемые форматы:

* PNG  
* JPG  
* PDF

  #### **4.3. Редактор шаблона**

Админ должен иметь возможность настроить:

* место, где вставляется имя  
* размер шрифта  
* шрифт  
* цвет текста  
* положение текста на сертификате

Желательно:

* превью сертификата  
* автоматическое уменьшение шрифта для длинных имен

  #### **4.4. Загрузка базы участников**

Админ загружает таблицу участников в формате:

* CSV  
* XLSX

Минимально обязательные поля:

* email  
* full\_name

Логика:

* система сверяет введенный пользователем e-mail с таблицей  
* если совпадение найдено, берет full\_name  
* подставляет его в сертификат

  #### **4.5. Управление выдачей**

Админ может:

* загрузить новую базу  
* выбрать шаблон  
* включить или выключить выдачу сертификатов  
* при необходимости заменить шаблон или список участников

  ### **5\. Безопасность**

Обязательно:

* админ-панель должна быть защищена логином и паролем  
* все защищенные роуты должны быть закрыты через **route protection**  
* доступ к админским роутам только при наличии валидного **JWT token**  
* должна быть реализована проверка роли пользователя для доступа к админ-функциям  
* пользователь не должен иметь доступ к базе участников  
* пользователь не должен видеть чужие данные  
* имя для сертификата должно подтягиваться только из базы, а не вводиться вручную  
* все проверки должны происходить на сервере  
* защита от подбора e-mail через rate limit / captcha  
* база данных не должна быть доступна с фронта

Желательно:

* отдельный поддомен для админки  
   например: admin...  
* HTTPS  
* ограничение попыток входа в админку  
* логирование действий администратора  
* refresh token механизм или безопасное обновление сессии  
* истечение JWT token по времени

  ### **6\. Архитектура и инфраструктура**

Нужно предусмотреть:

* backend, который обрабатывает проверку e-mail, генерацию сертификатов и работу с шаблонами  
* отдельную админскую часть с защищенными роутами  
* хранение шаблонов и таблиц участников в защищенном storage  
* генерацию сертификатов на сервере

Также необходимо настроить **Cluster** для стабильной работы системы при высокой нагрузке и одновременных запросах от большого количества пользователей.

Цель cluster-настройки:

* обеспечить устойчивость системы  
* распределять нагрузку  
* избежать падений при массовой выдаче сертификатов  
* поддерживать быструю обработку запросов в пиковые периоды

  ### **7\. Дополнительные функции**

Желательно добавить:

* уникальный ID сертификата  
* QR-код для проверки сертификата  
* страницу верификации сертификата  
* статистику: сколько сертификатов скачано  
* логи генерации  
* поддержку нескольких шаблонов под разные треки или категории участников

  ### **8\. Роли**

**Пользователь**

* вводит e-mail  
* получает сертификат  
* скачивает сертификат

**Администратор**

* входит в админку  
* загружает шаблоны  
* редактирует шаблон  
* загружает таблицу участников  
* включает и выключает выдачу

  ### **9\. Требования к результату**

Система должна позволять:

* быстро выдавать сертификаты для 5000+ участников  
* не делать сертификаты вручную  
* переиспользовать платформу для будущих хакатонов и программ

  ### **10\. MVP**

Минимальная версия должна включать:

* страницу ввода e-mail  
* проверку e-mail по базе  
* генерацию сертификата  
* скачивание PDF  
* админ-логин  
* JWT token authentication  
* protection of admin routes  
* загрузку шаблона  
* настройку позиции имени  
* загрузку CSV/XLSX с участниками  
* включение и выключение выдачи  
* базовую cluster-настройку для нагрузки
